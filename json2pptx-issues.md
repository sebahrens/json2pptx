# json2pptx Table Styling Issues

Tables generated by json2pptx cannot match template-styled tables because the engine always emits explicit XML elements that override the template's table style definitions. The fancy (manually styled) PPTX relies purely on Brand Table 3 (`{B56CA0EE-1E45-4AA6-A82B-0F5650326712}`) with `firstRow="1" bandRow="1"` and **zero** explicit borders or cell fills. json2pptx always adds both, making it impossible to let the table style drive appearance.

---

## Issue 1: `tblBorders` always emitted, even with `"borders": "none"`

**Current behavior:**
- `"borders": "horizontal"` â†’ emits `<a:tblBorders>` with `insideH`/`top`/`bottom` as `solidFill schemeClr=tx1`, rest as `noFill`
- `"borders": "none"` â†’ emits `<a:tblBorders>` with ALL borders set to `<a:ln><a:noFill/></a:ln>`
- `"borders": "all"` â†’ emits `<a:tblBorders>` with all borders as `solidFill schemeClr=tx1`
- No style at all â†’ emits `<a:tblBorders>` with all borders as `solidFill schemeClr=tx1`

**Problem:** Even `"borders": "none"` emits explicit `<a:tblBorders>` with `<a:noFill/>` entries. These explicit noFill elements **override** the table style's border definitions. Brand Table 3 defines `insideH` as `bg2`-colored 12700-width horizontal lines â€” but the explicit noFill suppresses them.

**Expected behavior:** When `"borders": "none"`, do NOT emit `<a:tblBorders>` at all. Let the table style's `wholeTbl > tcBdr` handle borders.

**Suggested fix in Go:** In the table rendering code, when `borders == "none"` (or when borders is omitted and a table style is being used), skip the `tblBorders` XML block entirely rather than writing noFill entries.

**XML comparison:**

json2pptx output (borders=none):
```xml
<a:tblPr firstRow="1" bandRow="1">
  <a:tblBorders>
    <a:top><a:ln w="0"><a:noFill/></a:ln></a:top>
    <a:bottom><a:ln w="0"><a:noFill/></a:ln></a:bottom>
    <a:left><a:ln w="0"><a:noFill/></a:ln></a:left>
    <a:right><a:ln w="0"><a:noFill/></a:ln></a:right>
    <a:insideH><a:ln w="0"><a:noFill/></a:ln></a:insideH>
    <a:insideV><a:ln w="0"><a:noFill/></a:ln></a:insideV>
  </a:tblBorders>
</a:tblPr>
```

Fancy (manual) output â€” no tblBorders at all:
```xml
<a:tblPr firstRow="1" bandRow="1"/>
```

## Issue 2: Header cells always get explicit `solidFill`, even without `header_background`

**Current behavior:** Every header cell (`firstRow`) receives an explicit `<a:solidFill><a:schemeClr val="accent1"/></a:solidFill>` in its `<a:tcPr>`, regardless of whether `header_background` is specified in JSON.

Tested all three variants:
- `"header_background": "accent1"` â†’ explicit accent1 solidFill on each header cell
- No `header_background` in style â†’ explicit accent1 solidFill on each header cell (same!)
- No style object at all â†’ explicit accent1 solidFill on each header cell (same!)

**Problem:** The explicit cell fill overrides the table style's `firstRow > tcStyle > fill`. Brand Table 3 defines firstRow fill as `bg1 lumMod=95000` (light gray) with accent1-colored text â€” a completely different look (orange text on gray) versus what json2pptx produces (white text on orange fill).

**Expected behavior:** When `header_background` is NOT specified, do NOT emit `<a:solidFill>` on header cells. Let the table style's `firstRow` definition handle header appearance.

**Suggested fix in Go:** Only emit the `<a:solidFill>` element inside header cell `<a:tcPr>` when `header_background` is explicitly set in the JSON input. When omitted, leave `<a:tcPr>` without a fill child.

**XML comparison:**

json2pptx output (no header_background specified):
```xml
<a:tc>
  <a:tcPr>
    <a:solidFill><a:schemeClr val="accent1"/></a:solidFill>
  </a:tcPr>
  <!-- ... -->
</a:tc>
```

Fancy (manual) output â€” no fill on header cells:
```xml
<a:tc>
  <a:tcPr/>
  <!-- ... -->
</a:tc>
```

## Issue 3: Header text color always set to white

**Related to Issue 2.** Because json2pptx always applies an accent1 fill to header cells, it also sets header text to white (`#FFFFFF`). When the fill is removed (Issue 2 fix), the text color should also defer to the table style.

Brand Table 3's `firstRow > tcTxStyle` sets text to `schemeClr val="accent1"` (orange, bold). json2pptx overrides this with explicit white text runs.

**Expected behavior:** When `header_background` is NOT specified, do not force white text on header cells. Let the table style's `firstRow > tcTxStyle` control text color.

---

## Issue 4: No way to opt into "style-only" table rendering

**Current behavior:** There is no JSON option to say "use the template's table style as-is, don't add any explicit formatting."

**Suggested addition:** A new style option, e.g. `"borders": "style"` or a boolean `"use_table_style": true`, that tells json2pptx:
1. Do NOT emit `<a:tblBorders>`
2. Do NOT emit `<a:solidFill>` on any cells (header or body)
3. Do NOT override text colors on header rows
4. Still emit `firstRow="1"` and `bandRow="1"` in `<a:tblPr>` so the table style's conditional formatting activates
5. Still reference the correct `tableStyleId` from the template

This would let users rely entirely on the template's table style definition, which is the standard PowerPoint workflow.

---

## Issue 5: `bandRow` defaults to `"0"` when no style is specified

**Current behavior:** When the `style` object is omitted entirely from the table JSON, `bandRow` is set to `"0"` in `<a:tblPr>`. This disables row banding even if the table style defines `band2H` alternating fills.

**Expected behavior:** Default `bandRow` to `"1"` so that the table style's banding definitions take effect.

## Issue 6: Contrast auto-fix cannot be disabled or configured

**Current behavior:** json2pptx enforces a minimum WCAG contrast ratio of ~3.0 between text color and fill color. When the ratio is below this threshold, it silently replaces the text color with a calculated dark gray that achieves exactly ~3.03 ratio. There is no JSON option to disable or adjust this behavior.

**Problem:** The auto-fix produces different gray replacements per fill color, creating visual inconsistency. A deck designed with white text on branded accent fills ends up with 5 different shades of gray text instead of uniform white.

**Measured behavior with PwC template accent colors (white `#FFFFFF` text):**

|---|---|---|---|---|---|
| `accent1` | #FD5108 | 3.31 | No â€” passes | â€” | â€” |
| `accent2` | #FE7C39 | 2.57 | **Yes** | #525252 | 3.03 |
| `accent3` | #FFAA72 | 1.87 | **Yes** | #676767 | 3.03 |
| `accent4` | #A1A8B3 | 2.40 | **Yes** | #575757 | 3.02 |
| `accent5` | #B5BCC4 | 1.92 | **Yes** | #656565 | 3.04 |
| `accent6` | #CBD1D6 | 1.54 | **Yes** | #747474 | 3.03 |
| `#65686B` | #65686B | 5.71 | No â€” passes | â€” | â€” |
| `#FFDCCE` | #FFDCCE | 1.28 | No â€” warn only | â€” | â€” |

**Expected behavior:** Add a JSON option to disable contrast enforcement, e.g. `"contrast_check": false` at the shape or deck level. Designers sometimes intentionally use low-contrast text for visual hierarchy (e.g., subtle watermarks, decorative text).

**Suggested fix in Go:** Add an optional boolean field `contrast_check` (default `true` for backward compatibility). When set to `false`, skip the contrast calculation and text color replacement entirely.

## Issue 7: Scheme colors are auto-fixed, but hex colors are only warned

**Current behavior:** The contrast auto-fix applies differently depending on how the fill color is specified:
- **Scheme colors** (`accent1`â€“`accent6`, `lt1`, `lt2`, `dk1`, `dk2`): Text color is silently replaced with a calculated gray
- **Hex colors** (`#RRGGBB`): A warning is logged to stderr but the text color is preserved as-is

**Problem:** This inconsistency means the same visual fill produces different results depending on how it's specified in JSON. `"fill": "accent3"` with white text â†’ gray text (#676767). But `"fill": "#FFAA72"` (the same color) with white text â†’ white text preserved with a warning.

**Measured example:**
- `accent3` fill + `#FFFFFF` text: contrast 1.87 â†’ **auto-fixed to #676767**
- `#FFDCCE` fill + `#FFFFFF` text: contrast 1.28 â†’ **warn only, #FFFFFF preserved**

**Expected behavior:** Either apply the fix consistently to both scheme and hex colors, or (better) provide the `contrast_check` option from Issue 6 so users can control it explicitly.

**Suggested fix in Go:** Unify the behavior: apply the same logic (fix or warn) regardless of whether the fill is a scheme color reference or a hex literal. Ideally, combine with Issue 6's `contrast_check` flag.

## Issue 8: No support for theme color modifiers (lumMod / lumOff)

**Current behavior:** json2pptx resolves semantic fill names (`accent1`, `lt2`, etc.) to their theme hex values and writes `<a:srgbClr val="RRGGBB"/>` in the output XML. There is no way to specify luminance modifiers (`lumMod`, `lumOff`) that PowerPoint uses for tints and shades of theme colors.

**Problem:** A manually-styled PPTX uses theme references with modifiers like:
```xml
<a:solidFill>
  <a:schemeClr val="accent3">
    <a:lumMod val="20000"/>
    <a:lumOff val="80000"/>
  </a:schemeClr>
</a:solidFill>
```
This produces a 20% tint of accent3 that automatically adapts if the theme changes. json2pptx cannot produce this â€” users must pre-calculate the hex equivalent (`#FFEEE3` for accent3 at 20%/80%) and hardcode it.

**Common tints needed in consulting decks:**

| Base | lumMod/lumOff | Calculated Hex | Description |
|---|---|---|---|
| accent1 (#FD5108) | 20000/80000 | #FFDCCE | Light salmon tint |
| accent2 (#FE7C39) | 20000/80000 | #FFE5D7 | Light peach tint |
| accent3 (#FFAA72) | 20000/80000 | #FFEEE3 | Very light peach |
| accent6 (#CBD1D6) | 50000/â€” | #65686B | Dark gray shade |

**Expected behavior:** Support an extended fill syntax for theme color modifiers:
```json
{"color": "accent3", "lumMod": 20000, "lumOff": 80000}
```

**Suggested fix in Go:** When the fill value is an object with `color` + `lumMod`/`lumOff`, emit `<a:schemeClr>` with child `<a:lumMod>` and `<a:lumOff>` elements instead of resolving to hex. The existing object fill syntax (`{"color": "accent1", "alpha": 50}`) could be extended to include these optional fields.

## Issue 9: All colors emitted as hardcoded hex, not as theme references

**Current behavior:** Even when the JSON input uses semantic names like `accent1` or `lt2`, json2pptx resolves them to hex values and writes `<a:srgbClr val="RRGGBB"/>` in the output XML. The generated PPTX contains no `<a:schemeClr>` references.

**Problem:** Hex values are baked in and won't adapt when the template's theme changes. A manually-styled PPTX uses `<a:schemeClr val="accent1"/>` so that if the template switches from orange to blue branding, all accent1 elements update automatically. json2pptx output hardcodes `#FD5108` and stays orange regardless.

**XML comparison:**

json2pptx output (accent1 fill):
```xml
<a:solidFill>
  <a:srgbClr val="FD5108"/>
</a:solidFill>
```

Manual PPTX output (accent1 fill):
```xml
<a:solidFill>
  <a:schemeClr val="accent1"/>
</a:solidFill>
```

**Expected behavior:** When the input specifies a semantic color name (`accent1`â€“`accent6`, `lt1`, `lt2`, `dk1`, `dk2`, `hlink`, `folHlink`, `bg1`, `bg2`, `tx1`, `tx2`), emit `<a:schemeClr>` in the output XML instead of resolving to hex.

**Suggested fix in Go:** In the color resolution code, distinguish between semantic names and hex literals. For semantic names, emit `<a:schemeClr val="..."/>`. For hex values (`#RRGGBB`), continue emitting `<a:srgbClr val="..."/>`. This may need to be applied in multiple places: shape fills, text colors, border colors, table cell fills.
